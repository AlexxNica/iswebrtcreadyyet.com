PeerConnections

The basic building block for connecting to peers.

getUserMedia

The ability to request access to a user's webcam and microphone.

mediaConstraints

The ability to specify restricted video sizes. This is important for bandwidth control.

WebAudio Integration

WebAudio enables processing of raw audio. For WebRTC applications it's important that this capability not only exist, but that it's capable of being used with a MediaStream as a source.

Screen Sharing

The ability to request access to a MediaStream of the computer screen. This is crucial for feature parity with existing communication solutions.

dataChannels

Data channel support comes in two flavors:

1. Reliable means they are guaranteed to arrive and arrive in the correct order.
2. Unreliable means that order doesn't matter and dropped messages are acceptable.

Stream re-broadcasting

In order to support some of the more interesting topologies for routing data, we need to be able to take a MediaStream object from one peer and add it to another PeerConneciton. Without this, the only way of doing multi-user is a mesh network or a centralized server.

Excellent echo cancellation

This is the most objective item in this list. But 3 or 4 users should be able to use a service like Talky without headphones on and not experience feedback problems.
